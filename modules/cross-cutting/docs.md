# Documentation Best Practices

- Keep documentation concise, actionable, and aligned with current system behavior.
- Write for readers with varying context: explain intent, prerequisites, and expected outcomes.
- Prefer task-oriented structure (`what`, `when`, `how`, `verify`, `rollback/recovery` where relevant).
- Keep setup and usage steps explicit, reproducible, and environment-aware.
- Document assumptions, constraints, and known limitations when they affect safe operation.
- Keep examples realistic and copy-paste safe; avoid stale placeholders for critical values.
- Update docs in the same change set when behavior, configuration, or workflows change.
- Ensure operational docs include troubleshooting signals, common failure modes, and escalation paths.
- Keep references between rules, skills, commands, and README consistent and non-contradictory.
- Prefer stable terminology and naming across code, docs, and runbooks.
- Remove or revise obsolete guidance promptly; stale docs are production risk.
- Include verification steps so readers can confirm a change worked as intended.
- Capture high-impact decisions briefly to preserve context for future maintainers.
